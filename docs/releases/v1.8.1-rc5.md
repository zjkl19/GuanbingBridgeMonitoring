# v1.8.1-rc5

`v1.8.1-rc5` is a recovery candidate for full-month, high-frequency analysis
runs that exhaust memory while publishing figures or frequency-domain results.
It does not change the accepted `full + connect` sampling, cleaning,
statistics, plotting, or report interpretation. Stable `F:\Guanbing` and all
source ZIP trees remain outside this candidate's write boundary.

## Candidate changes

- Publishes compact JSON through a same-directory temporary file, then reopens
  and decodes the exact bytes before the file is moved into place. A path that
  exists on disk is no longer sufficient evidence that a status or manifest is
  complete.
- If a full analysis manifest cannot be encoded or written, publishes a small,
  valid `status=failed` manifest without the bulky artifact arrays. Downstream
  report checks must reject it rather than treating a truncated file as a
  successful run.
- When one analysis step fails with `memory_error`, deletes only figure windows
  created by that step and skips later analysis modules marked as high-memory.
  Existing GUI windows and later low-memory steps keep their previous behavior.
- Corrects the seventh constructor parameter of the acceleration and cable
  acceleration adapters to `AutoDetectFs`. The hidden, read-only
  `SaveFigures=true` property remains solely for compatibility because these
  pipelines always publish their formal figures.
- Adds a repository-operated, offline, fail-closed recovery utility for the Jiulongjiang May
  boundary: 15 cable-acceleration points run in separate Runner processes,
  both spectrum modules run in their own fresh processes, single-row workbooks
  are hashed and merged in configured order, and a composite manifest is built
  only from validated requests, configurations, artifacts and source records.
  The invalid original manifest is diagnostic evidence only and is never an
  input to recovery.

The recovery utility is an operator tool, not an end-user workbench screen and
not an embedded feature of the packaged EXE. If it is used on machine 133, its
exact bytes must be deployed separately into the isolated RC tree with a
SHA-256 receipt.

## Recovery boundary

- Do not rerun the first 12 already completed Jiulongjiang May modules unless
  their independently rebuilt evidence fails validation.
- Do not resume cable acceleration at point 11. The failed module did not
  publish a complete statistics workbook, so all 15 points must be recomputed
  and merged in the configured order.
- Each point recovery is sequential and process-isolated. The next point may
  start only after the previous one-row workbook and its artifacts have been
  validated and hashed.
- `accel_spectrum` and `cable_accel_spectrum` each require a complete fresh
  process. A composite successful manifest may be published only after the
  first-12 evidence, all 15 point recoveries, the merged workbook, both
  spectrum results, group-plot disposition and all artifact hashes pass.
- Jiulongjiang's exact acceptance contract is 88 report-facing formal figure
  stubs but 42 formal `plot.json` records. The remaining 46 are spectrum
  figures (16 acceleration `SpecFreq`, 15 cable `SpecFreq` and 15
  `CableForce`) from a writer that does not currently emit `plot.json`.
  These two counts are validated and reported separately.
- The current Jiulongjiang cable-force parameters are `rho=1`, `L=1` without
  an engineering-verification marker. rc5 records them as
  `placeholder_parameters` and sets `cable_force_engineering_valid=false`.
  The generated force figures prove only that the configured process was
  reproduced; their values and plots must not support an engineering cable-
  force conclusion until formal parameters are supplied and marked verified.
- All remote writes remain under `F:\Guanbing_v1.8.1-rc1`. Deployment to stable
  production, source-data mutation, CSV fallback, cache rebuilding, resampling
  and WIM processing are outside this recovery.
- The rc4 verified extracted-CSV cleanup remains available but is unrelated to
  this recovery and stays default-off. It still requires a dedicated
  preprocessing task and the exact confirmation token
  `DELETE_VERIFIED_EXTRACTED_CSV`; no rc5 analysis or recovery job may enable
  it implicitly. Source ZIPs, WIM/Excel files, unconfigured CSVs and validated
  MAT/meta cache evidence remain retained exactly as in rc4.
