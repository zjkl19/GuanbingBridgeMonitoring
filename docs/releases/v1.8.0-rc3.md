# v1.8.0-rc3

`v1.8.0-rc3` is a validated dev-branch candidate for the unified PySide6 bridge
health-monitoring workbench. The rebuilt EXE, release inventory and disposable
install/update/rollback cycle pass. This is not a declaration that every bridge
has a newly generated and approved report, nor authorization to replace the
legacy production installation or merge `main`.

## Candidate changes

- Ordinary GUI data loading defaults to `auto`: use a valid CSV when present,
  otherwise use a valid compatible MAT cache. Users do not need to select
  `mat_only` just because an archived period contains no CSV files.
- Explicit `mat_only` remains an isolation/verification mode. `DataIndex` now
  honors that contract and no longer falls back to CSV while building its
  source inventory.
- Jiulongjiang and Shuixianhua daily exports can read existing `jlj_csv_v2`
  cache payloads (`ts`, `valx`, `valy`, `valz` and metadata) when CSV is absent.
  CSV behavior and CSV priority in `auto` remain unchanged.
- Zhishan and Shuixianhua cable-acceleration raw time-history figures use
  automatic y-limits. This does not change RMS figure limits, RMS statistics,
  cleaning or alarm interpretation.
- Report generation is embedded in the same workbench runtime and executes as
  a background worker started by `桥梁健康监测工作台.exe`. The unified package
  no longer includes or pops up a separate `BridgeReportBuilder.exe`.

  timestamp at `2026-06-08 09:00:02.057`; it first appears in the June 9
  export. Point `33` is temperature and `33-2` is highly likely relative
  humidity. Within the inspected export range, no independent `WS-H` channel
  was observed; authoritative vendor channel documentation is still absent.
- Zhishan April has no `2026-04-02` partition, lacks CF-7 on
  `2026-04-01`, and lacks the adjacent `2026-05-01` rolling source needed to
  reconstruct the April tail.
- Shuixianhua cable-acceleration data are absent from `2026-05-02` through

  fabricated.
