# v1.8.1-rc1

`v1.8.1-rc1` is an isolated release candidate for validating the
Jiulongjiang/Shuixianhua daily-export preprocessing workflow before any
production upgrade.

## Candidate changes

- Adds guarded batch extraction for daily ZIP exports. Source archives are
  retained, every file is checked against the archive index and CRC, unsafe or
  colliding Windows paths are rejected, and a same-volume staging directory is
  published only after verification succeeds.
- Separates the archive source root from the extraction output root so a
  candidate run can remain isolated from the original data tree.
- Adds a volume-space gate, output-root lock, extraction summary and reusable
  per-day manifest. The gate accounts for uncompressed bytes, filesystem
  overhead, planned downstream writes and a configured free-space reserve.
  Absolute `preprocessing.unzip.summary_file` paths are used unchanged; relative
  paths are resolved under the extraction output root.
- Adds the user-facing cache-prebuild module for Jiulongjiang/Shuixianhua.
  It reads extracted CSV files and creates or reuses raw MAT parse caches
  without cleaning, filtering, downsampling, moving or deleting source data.
- Adds an independent cache-space gate and per-file cache provenance summary.
  Analysis and cache prebuilding share the same CSV parser and cache contract.
- Keeps verified daily extraction directories reusable after downstream MAT
  caches are created: reuse checks remain bound to every ZIP-declared path and
  byte count while ignoring additional derived cache files. Missing or changed
  archive entries still fail closed.
- Fixes Windows ZIP directory entries that use trailing backslashes, explicit
  Jiulongjiang vendor/layout inference for isolated run directories, and async
  jobs incorrectly appearing completed when their final analysis manifest
  failed. A failed manifest is now persisted into the async status first, then
  the compiled Runner exits non-zero; packaging requires the real compiled
  failure-exit smoke recorded as `analysis_runner_failure_exit_smoke=true`.
- Jiulongjiang monthly reporting now treats traffic/WIM as explicitly not
  applicable when the bridge configuration declares it optional, while still
  failing closed for stale periods, images, statistics or report sources.
- Jiulongjiang patrol attachments are now period-bound and policy-driven.
  When `reporting.patrol.required=false` and no matching patrol source is
  available, the generator removes stale template patrol content, inserts an
  explicit current-period availability note, and records `not_available` plus
  the source decision in the report manifest without weakening other strict
  source checks. Required or explicitly mismatched patrol sources still fail.

## Release-candidate boundary

The 133-machine trial writes only to a separate candidate tree. The one-day
extraction/reuse and cache-idempotency gates have completed; the full May
extraction is currently running after a fresh capacity check. Full-period cache,
analysis and report acceptance are not complete. The existing production
application and original ZIP tree must not be overwritten.
