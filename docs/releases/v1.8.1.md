# v1.8.1

`v1.8.1` is the first stable maintenance release after the unified PySide6
workbench entered production. It keeps the validated MATLAB analysis kernel and
the existing statistical definitions, while closing the large-data,
cache-lifecycle and report-safety issues found during the v1.8.1 release-candidate
trials.

## Operator-visible changes

- Raw CSV cache prebuild is available to every configured bridge profile. For
  the three supported archive-backed layouts -- `jlj_daily_export`,
  `dated_folders` and `hongtang_period` -- an operator may also request daily
  deletion of the extracted CSV files that were actually consumed by the
  current configuration.
- Verified CSV deletion is disabled by default, is valid only for the current
  preprocessing task, and requires the exact confirmation token
  `DELETE_VERIFIED_EXTRACTED_CSV`. It is not a recursive cleanup tool. Original
  ZIP archives, WIM, Excel, unconfigured CSVs, MAT/meta cache pairs and audit
  receipts are retained.
- Each natural day is one transaction: the program verifies independently
  loadable MAT/meta pairs, unchanged and readable recovery ZIPs, unique
  relative-path/size/CRC matches and the durable receipt before deleting any CSV
  for that day. Missing or ambiguous evidence makes the whole day delete zero
  files. Cache workers never delete source files; only the parent process may
  commit a verified day.
- Report generation remains embedded in the same workbench executable. The
  workbench starts the report worker in the background and keeps the bound
  configuration, template, analysis result and quality-check state visible in
  one task history; a second report-builder GUI is not shipped.

## Large-data and analysis resilience

- Monthly high-frequency statistics continue to use the complete cleaned source
  series. Formal plot rendering uses bounded, peak-preserving vertices so a
  report image no longer requires another full-size plotting copy in memory.
  Batch figures stay hidden while rendering, but saved `.fig` files explicitly
  reopen as visible in an interactive MATLAB session.
- A memory failure now closes only figures created by the failed step and stops
  later high-memory modules from predictably failing in sequence. Recovery can
  isolate cable-acceleration points and spectrum modules in fresh Runner
  processes without changing the accepted `full + connect` calculation or
  plotting meaning.
- The permanent high-frequency path was compared with real Hongtang Q2 and
  Zhishan April data. Full-source statistics remained unchanged and accepted
  key figures retained the dense, continuous visual appearance of the reviewed
  baselines.
- Zhishan April dynamic-strain high-pass and low-pass processing was rerun as a
  separate real-data regression. Both statistics workbooks matched the accepted
  baseline cell-for-cell, all 32 reviewed JPG files matched their baseline
  hashes and sampled saved figures reopened normally. This independent run
  closes the dynamic-strain scope that was not part of the four-module
  high-frequency regression directory.

## Result and report integrity

- Status and analysis-result manifests are written as compact JSON through a
  same-directory temporary file and decoded again before atomic publication. If
  a full success manifest cannot be encoded or written, the Runner publishes a
  small valid `status=failed` manifest instead of leaving truncated JSON that a
  report could mistake for a completed run.
- Jiulongjiang cable-force conversion currently uses placeholder `rho`/`L`
  values. A strict monthly report includes engineering cable-force images and
  conclusions only when its pinned analysis manifest explicitly declares the
  Boolean `cable_force_engineering_valid=true`. A missing, false or non-Boolean
  value fails closed: the report retains measured cable-vibration analysis,
  but omits force images, force values and engineering force conclusions and
  discloses why.
- Release builds and GitHub publication are bound to an exact Git source commit.
  The build/release manifests record the source commit, and packaging rejects a
  dirty or mismatched source, an incomplete file inventory, missing compiled
  Runner/embedded-report checks, or a release-note/version mismatch.
- The release-health runner continuously drains subprocess output and uses a
  bounded post-exit collection window, so a completed MATLAB descendant that
  retains inherited Windows pipe handles cannot leave the release gate waiting
  indefinitely.

## Compatibility and safety boundaries

- Existing bridge configuration files, MAT cache schemas and analysis module
  keys remain compatible. Normal users should leave data mode on
  `auto`; it reads CSV when present and a valid MAT cache when verified CSV
  cleanup has removed the extracted source copy. `mat_only` remains an
  isolation and acceptance option.
- Safe CSV cleanup requires recoverable ZIP evidence for the exact extracted
  files. Data without an unchanged, readable and uniquely matching archive is
  never eligible for deletion, even if a MAT file exists.
- High-frequency memory changes do not alter cleaning thresholds, filtering,
  sampling-rate interpretation, statistics or report conclusions. Any future
  numerical change requires its own engineering review and regression.
- The Jiulongjiang May acceptance report has passed its independent Microsoft
  Word field update, authoritative PDF export and all-page review. This source
  release note still does not claim that machine 133 stable production has
  been switched to v1.8.1; that operational outcome is recorded only after the
  final package-hash, rollback, installation and post-install smoke gates pass.

## Required promotion checks

Before publishing or replacing a production installation:

1. Run the complete Python and MATLAB suites plus the focused cache-cleanup,
   large-manifest, high-frequency and monthly-report regressions.
2. Rebuild the workbench and compiled Runner from a clean, committed source;
   verify the closed per-file inventory, embedded report worker, all-profile
   matrix, CLI error path and native Windows font/icon/DPI/focus screenshots.
3. Generate each target report from its exact completed analysis result, update
   fields with Microsoft Word, export the authoritative PDF and inspect every
   page. Period text, media axes, captions, page fields and source disclosures
   must agree.
4. Deploy to an isolated directory first, rehash every file and run CLI/GUI/
   report smokes. Back up the stable application and retain a tested rollback
   path before the final cutover.
